TEIL 6 – Objektorientierte Programmierung (Kapitel 40–45)

Kapitel 40 – Objekte

Die Objektorientierte Programmierung (OOP) ist das zentrale Paradigma moderner Softwareentwicklung. Die Grundidee: Alles wird als Objekt modelliert – ähnlich wie in der realen Welt. Ein Auto hat Eigenschaften (Farbe, Marke, Kilometerstand) und Fähigkeiten (fahren, bremsen, tanken).

In Java gibt es zwei wichtige Konzepte:

•	Klasse = der Bauplan. Beschreibt, welche Daten (Attribute) und Methoden ein Objekt hat. Nur eine Definition, kein konkretes Ding.

•	Objekt = die konkrete Instanz des Bauplans. Aus einer Klasse können beliebig viele Objekte erzeugt werden.

Analogie:

Klasse  = Bauplan eines Hauses
Objekt  = das tatsächlich gebaute Haus

Codebeispiel (HelloTester.java) – erstes Objekt benutzen:

[CODE]
public class HelloTester {
    public static void main( String[] args ) {
        // Objekt vom Typ String erzeugen
        String begruessung = new String("Hallo Welt!");

        // Methoden auf dem Objekt aufrufen
        System.out.println( begruessung.toUpperCase() );
        System.out.println( begruessung.length() );
        System.out.println( begruessung.replace("Welt", "Java") );
    }
}
[/CODE]

Ausgabe:

HALLO WELT!
11
Hallo Java!

Erklärung zum Code:

new String("Hallo Welt!") erzeugt ein neues Objekt der Klasse String im Speicher. Die Variable begruessung hält die Referenz (Adresse) auf dieses Objekt. Über den Punkt-Operator . werden Methoden auf dem Objekt aufgerufen. Das Objekt selbst enthält die Daten (den Text) und die Methoden, die darauf arbeiten.

Aufbau einer Klasse – Überblick:

[CODE]
public class Klasse {
    // Attribute (Daten des Objekts)
    int attribut;

    // Konstruktor (zum Erzeugen von Objekten)
    public Klasse( int wert ) {
        this.attribut = wert;
    }

    // Methoden (Fähigkeiten des Objekts)
    public void tuEtwas() {
        System.out.println("Attributwert: " + attribut);
    }
}
[/CODE]

Merksätze:

•	Klasse = Bauplan, Objekt = konkrete Instanz.
•	new erzeugt ein Objekt im Heap-Speicher.
•	Der Punkt-Operator . greift auf Methoden und Attribute eines Objekts zu.
•	Aus einer Klasse können beliebig viele Objekte erzeugt werden.





Kapitel 41 – Objekt-Referenzen

In Java speichern Variablen von Objekttypen keine Kopie des Objekts – sie speichern eine Referenz, also eine Speicheradresse, die auf das Objekt zeigt. Das ist ein fundamentaler Unterschied zu primitiven Datentypen wie int, die ihren Wert direkt speichern.
Primitiv vs. Referenz – der Unterschied:


// Primitiv: jede Variable hat ihre eigene Kopie
int a = 5;
int b = a;
b = 10;
System.out.println(a); // 5 – a bleibt unverändert

// Referenz: beide Variablen zeigen auf DASSELBE Objekt
String s1 = new String("Hallo");
String s2 = s1;         // s2 zeigt jetzt auf dasselbe Objekt wie s1

Codebeispiel (PointEg1.java / PointEg2.java) – Referenzen verstehen:

[CODE]
public class PointEg1 {
    public static void main( String[] args ) {
        // Zwei separate Objekte mit gleichem Inhalt
        String name1 = new String("Max");
        String name2 = new String("Max");

        System.out.println( name1 == name2 );       // false! verschiedene Adressen
        System.out.println( name1.equals(name2) );  // true! gleicher Inhalt
    }
}
[/CODE]


Erklärung zum Code:

== vergleicht bei Objekten die Speicheradressen (Referenzen), nicht den Inhalt. name1 und name2 zeigen auf zwei verschiedene Objekte – auch wenn der Text gleich ist. Deshalb ist == für Objekte fast immer falsch. .equals() vergleicht den Inhalt – das ist bei Strings (und fast immer bei Objekten) das Richtige.

Codebeispiel (PointEg3.java) – eine Referenz, ein Objekt:

[CODE]
public class PointEg3 {
    public static void main( String[] args ) {
        StringBuilder sb1 = new StringBuilder("Hallo");
        StringBuilder sb2 = sb1; // sb2 zeigt auf DASSELBE Objekt!

        sb2.append(" Welt");     // Änderung über sb2...

        System.out.println(sb1); // ...ist auch über sb1 sichtbar!
        // Ausgabe: Hallo Welt
    }
}
[/CODE]

Erklärung zum Code:

sb2 = sb1 kopiert nicht das Objekt, sondern nur die Referenz. Beide Variablen zeigen jetzt auf dasselbe StringBuilder-Objekt im Speicher. Eine Änderung über sb2 ist daher auch über sb1 sichtbar. Das ist ein häufiger Fehler bei Anfängern, der schwer zu debuggen ist.
null – die leere Referenz:


String name = null; // name zeigt auf kein Objekt

if ( name != null ) {
    System.out.println(name.length()); // sicher
}
// name.length() ohne null-Prüfung → NullPointerException!

null bedeutet: Die Variable existiert, zeigt aber auf kein Objekt. Der Aufruf einer Methode auf null ist ein NullPointerException-Fehler – einer der häufigsten Laufzeitfehler in Java.

Merksätze:
•	Objektvariablen speichern Referenzen (Adressen), keine Kopien.
•	== bei Objekten vergleicht Adressen, .equals() vergleicht Inhalte.
•	Mehrere Variablen können auf dasselbe Objekt zeigen → Änderung sichtbar für alle.
•	null = keine Referenz → Methode aufrufen → NullPointerException.







Kapitel 42 – Weitere Aspekte von Objekten und Klassen


Jetzt wird es konkret: Du schreibst deine eigenen Klassen. Eine vollständige Klasse besteht aus:

•	Attributen (auch: Instanzvariablen) – die Daten, die jedes Objekt besitzt.

•	Konstruktor – eine spezielle Methode, die beim Erzeugen mit new aufgerufen wird und das Objekt initialisiert.

•	Methoden – die Fähigkeiten des Objekts.

Codebeispiel (SimpleConstructor.java) – eigene Klasse mit Konstruktor:

[CODE]
public class SimpleConstructor {
    // Attribute
    String name;
    int    alter;

    // Konstruktor
    public SimpleConstructor( String name, int alter ) {
        this.name  = name;   // this.name = Attribut, name = Parameter
        this.alter = alter;
    }

    // Methode
    public void vorstellen() {
        System.out.println("Ich bin " + name + " und " + alter + " Jahre alt.");
    }

    // Tester (main-Methode)
    public static void main( String[] args ) {
        SimpleConstructor person1 = new SimpleConstructor("Anna", 22);
        SimpleConstructor person2 = new SimpleConstructor("Ben",  30);

        person1.vorstellen(); // Ich bin Anna und 22 Jahre alt.
        person2.vorstellen(); // Ich bin Ben und 30 Jahre alt.
    }
}
[/CODE]

Erklärung zum Code:

this.name bezieht sich auf das Attribut der Klasse. name (ohne this) ist der Parameter des Konstruktors. Da beide denselben Namen haben, muss this verwendet werden, um sie zu unterscheiden. person1 und person2 sind zwei unabhängige Objekte – jedes hat seinen eigenen name und alter.

Das Schlüsselwort this:

this ist eine Referenz auf das aktuelle Objekt. Es wird hauptsächlich genutzt, um:

1.	Attribut und Parameter mit gleichem Namen zu unterscheiden.

2.	Konstruktoren untereinander aufzurufen (this(...)).

Standardkonstruktor:

Schreibst du keinen Konstruktor, stellt Java automatisch einen leeren Standardkonstruktor bereit: public KlassenName() {}. Sobald du jedoch selbst einen Konstruktor schreibst, verschwindet der Standardkonstruktor – du musst ihn dann explizit hinzufügen, wenn du ihn noch brauchst.

Mehrere Objekte – unabhängig:

SimpleConstructor p1 = new SimpleConstructor("Clara", 25);
SimpleConstructor p2 = new SimpleConstructor("Dirk",  19);

// Jedes Objekt hat seine eigenen Daten:
System.out.println(p1.name); // Clara
System.out.println(p2.name); // Dirk

Typische Fehler:

•	this vergessen wenn Parameter und Attribut gleich heißen → Attribut wird nicht gesetzt.

•	Konstruktor-Name weicht vom Klassennamen ab → Compilerfehler.

•	void vor Konstruktor schreiben → Compilerfehler, Konstruktoren haben keinen Rückgabetyp.






Kapitel 43 – Methodenparameter


Methoden können Werte entgegennehmen (Parameter) und Werte zurückgeben (Rückgabewert). Das macht Methoden wiederverwendbar – statt Code zu kopieren, rufst du dieselbe Methode mit verschiedenen Werten auf.
Aufbau einer Methode:


public RückgabeTyp methodenName( TypParam1 param1, TypParam2 param2 ) {
    // Berechnung
    return ergebnis; // nur wenn kein void
}

Codebeispiel (OwnMethod.java):

[CODE]
public class OwnMethod {

    // Methode mit Parametern und Rückgabewert
    public static int addiere( int a, int b ) {
        return a + b;
    }

    // Methode ohne Rückgabewert (void)
    public static void begruessung( String name ) {
        System.out.println("Hallo, " + name + "!");
    }

    public static void main( String[] args ) {
        int summe = addiere(10, 25);
        System.out.println("Summe: " + summe); // 35

        begruessung("Max");   // Hallo, Max!
        begruessung("Lisa");  // Hallo, Lisa!
    }
}
[/CODE]

Erklärung zum Code:

addiere(10, 25) übergibt die Werte 10 und 25. Innerhalb der Methode heißen sie a und b – das sind lokale Variablen, die nur innerhalb dieser Methode existieren. return a + b gibt das Ergebnis zurück. Der Rückgabewert wird in der aufrufenden Methode in summe gespeichert. void bedeutet: kein Rückgabewert.

Codebeispiel (AddSubtract.java) – mehrere Methoden kombiniert:

[CODE]
public class AddSubtract {

    public static double addiere( double a, double b ) {
        return a + b;
    }

    public static double subtrahiere( double a, double b ) {
        return a - b;
    }

    public static void main( String[] args ) {
        double x = 15.5;
        double y = 4.5;

        System.out.println("Addition:    " + addiere(x, y));      // 20.0
        System.out.println("Subtraktion: " + subtrahiere(x, y));  // 11.0
        System.out.println("Verkettet:   " + addiere(x, subtrahiere(y, 1.0))); // 19.0
    }
}
[/CODE]

Erklärung zum Code:

Methoden können als Argumente für andere Methoden verwendet werden – subtrahiere(y, 1.0) wird zuerst ausgewertet (3.5), das Ergebnis wird dann an addiere übergeben (15.5 + 3.5 = 19.0). Java wertet Ausdrücke immer von innen nach außen aus.

Wertübergabe bei primitiven Typen (Call by Value):

Bei primitiven Typen (int, double, boolean etc.) wird eine Kopie des Wertes übergeben. Änderungen innerhalb der Methode haben keinen Einfluss auf die Original-Variable.

[CODE]
public static void verdoppeln( int zahl ) {
    zahl = zahl * 2;  // ändert nur die LOKALE Kopie
    System.out.println("In der Methode: " + zahl); // 10
}

public static void main( String[] args ) {
    int x = 5;
    verdoppeln(x);
    System.out.println("Danach: " + x); // 5 – x wurde NICHT geändert!
}
[/CODE]

Referenzübergabe bei Objekten:

Bei Objekten wird die Referenz (Speicheradresse) übergeben. Die Methode arbeitet direkt am Original-Objekt. Änderungen am Objekt selbst sind nach dem Methodenaufruf sichtbar.

[CODE]
public static void anhaengen( StringBuilder sb ) {
    sb.append(" Welt");  // ändert das ORIGINALOBJEKT
}

public static void main( String[] args ) {
    StringBuilder text = new StringBuilder("Hallo");
    anhaengen(text);
    System.out.println(text); // Hallo Welt – Original wurde geändert!
}
[/CODE]

Codebeispiel (TriangleTester.java) – Methode mit Objekt-Parameter:

[CODE]
public class TriangleTester {

    public static double flaeche( double basis, double hoehe ) {
        return 0.5 * basis * hoehe;
    }

    public static double umfang( double a, double b, double c ) {
        return a + b + c;
    }

    public static void main( String[] args ) {
        double basis = 6.0;
        double hoehe = 4.0;
        double seiteA = 5.0, seiteB = 5.0, seiteC = 6.0;

        System.out.printf("Fläche:  %.2f%n", flaeche(basis, hoehe));         // 12.00
        System.out.printf("Umfang:  %.2f%n", umfang(seiteA, seiteB, seiteC)); // 16.00
    }
}
[/CODE]

Lokale Variablen vs. Attribute:


Merkmal			Lokale Variable				Attribut (Instanzvariable)

Wo deklariert		Innerhalb einer Methode			Innerhalb der Klasse, außerhalb von Methoden
Gültigkeitsbereich	Nur in der Methode			Für alle Methoden der Klasse
Lebensdauer		Nur während Methodenaufruf		Solange das Objekt existiert
Initialwert		Keiner (muss manuell gesetzt werden)	Automatisch (0, false, null)

Merksatz:

Primitiv → Kopie übergeben → Original bleibt unverändert
Objekt   → Referenz übergeben → Original kann verändert werden

Typische Fehler:

•	Rückgabewert vergessen: return fehlt obwohl kein void → Compilerfehler.

•	Falscher Rückgabetyp: Methode deklariert int, gibt aber double zurück → Compilerfehler oder Informationsverlust.

•	Lokale Variable mit gleichem Namen wie Attribut ohne this → Attribut wird nicht gesetzt, lokale Variable wird verwendet (stille Falle).

•	Ergebnis von return nicht speichern: addiere(3, 4); ohne Zuweisung → Ergebnis geht verloren.






Kapitel 44 – Strings und Objekt-Referenzen


String ist in Java eine Klasse, kein primitiver Typ. Wie bei allen Objekten speichert eine String-Variable nur eine Referenz auf das Objekt im Speicher – nicht den Text selbst. Das hat eine wichtige Konsequenz: Der Operator == vergleicht bei Strings nicht den Inhalt, sondern die Speicheradressen.
Das Problem mit ==:


String a = new String("Hallo");
String b = new String("Hallo");

System.out.println(a == b);        // false! verschiedene Objekte, verschiedene Adressen
System.out.println(a.equals(b));   // true!  gleicher Inhalt

a und b sind zwei separate Objekte im Speicher – auch wenn beide den Text „Hallo" enthalten. == schaut auf die Adresse, nicht den Inhalt. Deshalb: Bei Strings (und allen Objekten) immer .equals() verwenden.

Codebeispiel (EqualsDemo.java):

[CODE]
public class EqualsDemo {
    public static void main( String[] args ) {
        String eingabe = "java";
        String ziel    = "Java";

        // Vergleich mit ==
        if ( eingabe == ziel ) {
            System.out.println("== : gleich");
        } else {
            System.out.println("== : NICHT gleich");  // ← wird ausgegeben
        }

        // Vergleich mit equals()
        if ( eingabe.equals(ziel) ) {
            System.out.println("equals: gleich");
        } else {
            System.out.println("equals: NICHT gleich"); // ← wird ausgegeben
        }

        // Vergleich ohne Groß-/Kleinschreibung
        if ( eingabe.equalsIgnoreCase(ziel) ) {
            System.out.println("equalsIgnoreCase: gleich"); // ← wird ausgegeben
        }
    }
}
[/CODE]

Ausgabe:

== : NICHT gleich
equals: NICHT gleich
equalsIgnoreCase: gleich

Erklärung zum Code:

"java" und "Java" unterscheiden sich durch die Groß-/Kleinschreibung. .equals() beachtet das – sie sind nicht gleich. .equalsIgnoreCase() ignoriert Groß-/Kleinschreibung – hier sind sie gleich. Das ist in der Praxis wichtig, z.B. wenn Benutzereingaben geprüft werden, die der Nutzer in beliebiger Schreibweise eintippen kann.

Codebeispiel (EqualsDemo2.java) – String-Literale und der String-Pool:

[CODE]
public class EqualsDemo2 {
    public static void main( String[] args ) {
        // String-Literale: Java speichert sie im String-Pool
        String s1 = "Hallo";
        String s2 = "Hallo";

        System.out.println(s1 == s2);       // true! (gleiche Adresse im Pool)
        System.out.println(s1.equals(s2));  // true!

        // Mit new: immer neues Objekt, außerhalb des Pools
        String s3 = new String("Hallo");
        System.out.println(s1 == s3);       // false! (andere Adresse)
        System.out.println(s1.equals(s3));  // true!
    }
}
[/CODE]

Erklärung zum Code:

Java optimiert String-Literale (also Texte direkt in Anführungszeichen) mit einem internen String-Pool: Gleiche Literale teilen sich denselben Speicherplatz. Deshalb ist s1 == s2 hier ausnahmsweise true. Mit new String(...) wird hingegen immer ein neues Objekt erzeugt, das außerhalb des Pools liegt – s1 == s3 ist daher false. Das macht == bei Strings noch unzuverlässiger: Es hängt davon ab, ob Java den Pool nutzt oder nicht. Fazit: Niemals == für String-Vergleiche verwenden.

Codebeispiel (EqualsICTester.java) – sicherer Vergleich in der Praxis:

[CODE]
import java.util.Scanner;

public class EqualsICTester {
    public static void main( String[] args ) {
        Scanner scanner = new Scanner(System.in);

        System.out.print("Passwort eingeben: ");
        String eingabe = scanner.next();

        if ( eingabe.equals("Geheim123") ) {
            System.out.println("Zugang gewährt.");
        } else {
            System.out.println("Falsches Passwort.");
        }
    }
}
[/CODE]

Sonderfall – compareTo():

Neben equals() gibt es noch compareTo(), das zwei Strings lexikografisch (wie im Wörterbuch) vergleicht:


String a = "Anna";
String b = "Berta";

int ergebnis = a.compareTo(b);

if      ( ergebnis < 0 ) System.out.println(a + " kommt vor " + b);
else if ( ergebnis > 0 ) System.out.println(a + " kommt nach " + b);
else                     System.out.println("Gleich");
// Ausgabe: Anna kommt vor Berta

compareTo() gibt zurück:

•	negativen Wert → erster String kommt alphabetisch vor dem zweiten

•	positiven Wert → erster String kommt nach dem zweiten

•	0 → beide sind gleich
Das wird vor allem beim Sortieren von Strings verwendet.

Merksätze:

•	== bei Strings → vergleicht Adressen → fast immer falsch verwenden.

•	.equals() → vergleicht Inhalte → immer verwenden.

•	.equalsIgnoreCase() → vergleicht ohne Groß-/Kleinschreibung.

•	.compareTo() → alphabetischer Vergleich, Ergebnis < 0, 0, oder > 0.

•	String-Literale liegen im String-Pool, new String(...) erzeugt ein neues Objekt.

Typische Fehler:

•	if (eingabe == "ja") → funktioniert zufällig manchmal (String-Pool), aber ist falsch und gefährlich.

•	.equals() auf null aufrufen → NullPointerException. Sicherer: "Konstante".equals(variable) – dann kann die Variable null sein, ohne Fehler.


// Unsicher:
if ( eingabe.equals("ja") )       // NullPointerException wenn eingabe == null

// Sicher (Konstante links):
if ( "ja".equals(eingabe) )       // kein Fehler, auch wenn eingabe null ist




Kapitel 45 – Mehr über Strings

String ist eine der meistgenutzten Klassen in Java. Wichtig: Strings sind unveränderlich (immutable). Jede Methode, die einen String scheinbar „verändert", gibt einen neuen String zurück. Das Original bleibt immer unberührt.

Überblick der wichtigsten String-Methoden:

Methode			Bedeutung			Beispiel → Ergebnis

length()		Anzahl Zeichen			"Hallo".length() → 5
charAt(i)		Zeichen an Position i		"Hallo".charAt(1) → 'a'
substring(i)		Teilstring ab Index i		"Hallo".substring(2) → "llo"
substring(i, j)		Teilstring von i bis j (exkl.)	"Hallo".substring(1, 3) → "al"
indexOf(s)		1. Pos. v. s, -1 w. nicht gef.	"Hallo".indexOf("ll") → 2
contains(s)		enthält s?			"Hallo".contains("all") → true
toUpperCase()		alle Großbuchstaben		"hallo".toUpperCase() → "HALLO"
toLowerCase()		alle Kleinbuchstaben		"HALLO".toLowerCase() → "hallo"
trim()			Leerzeichen am Rand entfernen	" hi ".trim() → "hi"
replace(alt, neu)	Text ersetzen			"Hallo".replace("l","r") → "Harro"
startsWith(s)		beginnt mit s?			"Hallo".startsWith("Ha") → true
endsWith(s)		endet mit s?			"Hallo".endsWith("lo") → true
isEmpty()		ist leer?			"".isEmpty() → true
split(trennzeichen)	in Array aufteilen		"a,b,c".split(",") → ["a","b","c"]

Codebeispiel (StringDemo2.java) – grundlegende Methoden:

[CODE]
public class StringDemo2 {
    public static void main( String[] args ) {
        String text = "  Java ist toll!  ";

        System.out.println("Original:        '" + text + "'");
        System.out.println("Trimmed:         '" + text.trim() + "'");
        System.out.println("Großbuchstaben:   " + text.trim().toUpperCase());
        System.out.println("Länge (trimmed):  " + text.trim().length());
        System.out.println("Enthält 'toll':   " + text.contains("toll"));
        System.out.println("Ersetzt:          " + text.trim().replace("toll", "super"));
    }
}
[/CODE]

Ausgabe:

Original:        '  Java ist toll!  '
Trimmed:         'Java ist toll!'
Großbuchstaben:   JAVA IST TOLL!
Länge (trimmed):  14
Enthält 'toll':   true
Ersetzt:          Java ist super!

Erklärung zum Code:

text.trim() gibt einen neuen String ohne führende/nachfolgende Leerzeichen zurück – text selbst bleibt unverändert. Methoden können verkettet werden: text.trim().toUpperCase() → erst trimmen, dann in Großbuchstaben. Das nennt sich Method Chaining und ist ein sehr häufiges Muster in Java.

Codebeispiel (StringDemo3.java) – substring und indexOf:

[CODE]
public class StringDemo3 {
    public static void main( String[] args ) {
        String email = "max.mustermann@example.de";

        // Position des @-Zeichens finden
        int atPos = email.indexOf("@");
        System.out.println("@ an Position: " + atPos);  // 14

        // Benutzername (alles vor @)
        String benutzer = email.substring(0, atPos);
        System.out.println("Benutzer: " + benutzer);    // max.mustermann

        // Domain (alles nach @)
        String domain = email.substring(atPos + 1);
        System.out.println("Domain:   " + domain);      // example.de

        // Einfache Gültigkeitsprüfung
        if ( email.contains("@") && email.contains(".") ) {
            System.out.println("Gültige E-Mail-Struktur.");
        }
    }
}
[/CODE]

Erklärung zum Code:

indexOf("@") liefert den Index des @-Zeichens. substring(0, atPos) schneidet von Position 0 bis kurz vor dem @. substring(atPos + 1) nimmt alles nach dem @. Gibt indexOf() den Wert -1 zurück, wurde das Zeichen nicht gefunden – ein substring(-1) würde dann eine StringIndexOutOfBoundsException auslösen. In echtem Code immer vorher prüfen ob das Ergebnis von indexOf nicht -1 ist.


Codebeispiel – split() für strukturierte Daten (CSV):

[CODE]
public class SplitDemo {
    public static void main( String[] args ) {
        // CSV-Zeile (Comma Separated Values) – typisch in der Ausbildung
        String datensatz = "Müller,Hans,42,Berlin";
        String[] felder  = datensatz.split(",");

        System.out.println("Nachname: " + felder[0]); // Müller
        System.out.println("Vorname:  " + felder[1]); // Hans
        System.out.println("Alter:    " + felder[2]); // 42
        System.out.println("Stadt:    " + felder[3]); // Berlin
    }
}
[/CODE]

Erklärung zum Code:

split(",") zerlegt den String an jedem Komma und gibt ein String-Array zurück. Der Index felder[0] ist der erste Teil, felder[1] der zweite usw. Das ist ein Standardmuster beim Einlesen von CSV-Dateien, die in der Ausbildung und in echten Anwendungen sehr häufig vorkommen.

Strings sind unveränderlich – die häufigste Falle:


String text = "hallo welt";

// FALSCH: Ergebnis wird nicht gespeichert
text.toUpperCase();
System.out.println(text); // hallo welt – unverändert!

// RICHTIG: Rückgabewert speichern
text = text.toUpperCase();
System.out.println(text); // HALLO WELT

Jede String-Methode gibt einen neuen String zurück. Wenn du das Ergebnis nicht in einer Variable speicherst, ist es sofort verloren. Das ist einer der häufigsten Anfängerfehler in Java.

Codebeispiel – Zeichen für Zeichen durchlaufen:

[CODE]
public class ZeichenZaehlen {
    public static void main( String[] args ) {
        String wort         = "Programmierung";
        int    vokalZaehler = 0;
        String vokale       = "aeiouAEIOU";

        for ( int i = 0; i < wort.length(); i++ ) {
            char c = wort.charAt(i);
            if ( vokale.contains(String.valueOf(c)) ) {
                vokalZaehler++;
            }
        }

        System.out.println("Vokale in '" + wort + "': " + vokalZaehler); // 6
    }
}
[/CODE]

Erklärung zum Code:

charAt(i) liefert das Zeichen an Position i als char. String.valueOf(c) wandelt den char in einen einzeichigen String um, damit contains() genutzt werden kann. Die for-Schleife läuft von 0 bis wort.length() - 1, weil Indizes in Java immer bei 0 beginnen.

String vs. StringBuilder – wann was verwenden:

Klasse			Unveränderlich?		Verwendung
String			ja			Text lesen, vergleichen, analysieren
StringBuilder		nein			Text schrittweise aufbauen, z.B. in Schleifen

Wenn du in einer Schleife viele Strings zusammenbaust, ist StringBuilder deutlich effizienter – weil kein neues Objekt pro Schritt erzeugt wird. Mehr dazu in Kapitel 68.

Typische Fehler:

•	text.trim() oder text.toUpperCase() ohne Zuweisung → Text bleibt unverändert, kein Compilerfehler, aber falsches Ergebnis.

•	indexOf() gibt -1 zurück, dann substring() aufrufen → StringIndexOutOfBoundsException.

•	charAt(i) mit Index >= length() → StringIndexOutOfBoundsException.

•	Vergessen dass Indizes bei 0 beginnen: das erste Zeichen ist charAt(0), nicht charAt(1).

•	split() auf einem leeren String gibt ein Array mit einem leeren Element zurück, nicht ein leeres Array.

Kapitel 46 – Eigene Klassen definieren (Defining Your Own Classes)

Was ist Objektorientierte Programmierung?

OOP besteht aus drei Phasen:

1.	Programm schreiben – Der Entwickler erstellt Quellcode-Dateien, in denen er Klassen für Objekte definiert. Eine dieser Klassen enthält die statische main()-Methode, mit der das Programm startet.

2.	Kompilieren – Der Java-Compiler übersetzt den Quellcode in Bytecode. Jede Klasse ergibt eine eigene .class-Datei.

3.	Programm ausführen – Die Java Virtual Machine (JVM) sucht nach der main()-Methode und startet sie. Während der Ausführung werden Objekte erstellt und ihre Methoden aufgerufen. Das Programm erledigt seine Arbeit durch das Erstellen von Objekten und den Aufruf ihrer Methoden.
Ein laufendes Programm ist eine Sammlung kommunizierender Objekte – ähnlich wie Mitarbeiter in einem Betrieb, die miteinander interagieren, um eine gemeinsame Aufgabe zu erfüllen.


Syntax einer Klassendefinition

Eine Klassendefinition sieht in Java so aus:

public class KlassenName
{
    // Beschreibung der Instanzvariablen (Felder)

    // Beschreibung der Konstruktoren

    // Beschreibung der Methoden
}

•	Der Modifier ist zunächst immer public für die Hauptklasse (die main() enthält).

•	Die Gliederung in drei Abschnitte (Felder, Konstruktoren, Methoden) ist keine Pflicht, aber gute Praxis für Lesbarkeit.

•	Ein Programm kann mehrere Klassen enthalten – für eine einfache Klasse reichen wenige Zeilen, für komplexe Klassen können es Tausende sein.

Methoden definieren

Methoden werden innerhalb einer Klasse wie folgt definiert:

public RückgabeTyp methodenName( parameterListe )
{
    // Java-Anweisungen

    return rückgabeWert;
}

•	RückgabeTyp: Der Datentyp des Wertes, den die Methode zurückgibt. Mit return wird dieser Wert an den Aufrufer übergeben.

•	void: Wenn eine Methode keinen Wert zurückgibt, wird void als Rückgabetyp verwendet. Ein return; am Ende ist dann optional.

•	Methoden werden mit public deklariert, damit sie von anderen Klassen aufgerufen werden können.

Das erste vollständige OOP-Programm: HelloObject
Das klassische "Hello World" in OOP-Form – mit zwei Klassen:

[CODE]
// HelloObject.java
public class HelloObject
{
    // Methodendefinition
    public void speak()
    {
        System.out.println("Hello from an object!");
    }
}

// HelloTester.java
public class HelloTester
{
    public static void main( String[] args )
    {
        HelloObject anObject = new HelloObject();
        anObject.speak();
    }
}
[/CODE]

Ausgabe:

Hello from an object!

So läuft das Programm ab:

1.	main() startet.

2.	Ein HelloObject-Objekt wird mit new HelloObject() erstellt. Der Standard-Konstruktor wird verwendet.

3.	Die speak()-Methode des Objekts wird aufgerufen.

4.	Das Programm endet.

Die Driver-Klasse (Test-Klasse)

Es ist üblich, eine separate Klasse zu erstellen, deren einziger Zweck es ist, die main()-Methode zu enthalten und das Programm zu starten. Diese Klasse wird manchmal Driver oder Tester genannt. Sie erstellt Objekte der anderen Klassen und ruft deren Methoden auf. Die eigentliche Arbeit erledigen die anderen Objekte.
Beim Kompilieren entstehen für jede Klasse eigene .class-Bytecode-Dateien. Das Programm wird immer mit dem Namen der Klasse gestartet, die main() enthält:

javac HelloTester.java
java HelloTester

Instanzvariablen (Felder)

Um verschiedene Objekte mit unterschiedlichen Daten zu ermöglichen, braucht die Klasse Instanzvariablen. Jedes Objekt (jede Instanz) hat seine eigenen Kopien dieser Variablen. Sie sind üblicherweise private deklariert, damit kein direkter Zugriff von außen möglich ist.
Ein Objekt hat drei Eigenschaften:

•	Identität: Jedes Objekt ist einzigartig.

•	Zustand: Die aktuellen Werte seiner Instanzvariablen.

•	Verhalten: Die Methoden, die auf den Instanzvariablen operieren.

Konstruktoren

Um Instanzvariablen beim Erstellen eines Objekts zu initialisieren, braucht die Klasse einen Konstruktor.
Syntax eines Konstruktors:

public KlassenName( parameterListe )
{
    // Anweisungen zur Initialisierung der Instanzvariablen
}

Wichtige Regeln für Konstruktoren:

•	Der Konstruktor hat denselben Namen wie die Klasse.

•	Es gibt keinen Rückgabetyp (auch nicht void).

•	Kein return-Statement im Body.

•	Eine Klasse kann mehrere Konstruktoren mit unterschiedlichen Parameterlisten haben.

Wenn kein Konstruktor definiert wird, stellt der Java-Compiler automatisch einen Standard-Konstruktor (Default Constructor) bereit – dieser tut hinter den Kulissen aber viel: Er reserviert Speicher, legt die Instanzvariablen an und gibt eine Objektreferenz zurück.
Wichtige Regel: Sobald man selbst mindestens einen Konstruktor definiert, stellt Java den Standard-Konstruktor nicht mehr automatisch zur Verfügung.

Das Schlüsselwort this

Wenn ein Parametername denselben Namen wie eine Instanzvariable hat, entsteht eine Namenskollision. Mit this wird eindeutig auf die Instanzvariable des aktuellen Objekts verwiesen:

[CODE]
class HelloObject
{
    private String greeting;

    public HelloObject( String greeting )
    {
        this.greeting = greeting; // this.greeting = Instanzvariable
                                  // greeting       = Parameter
    }

    public void speak()
    {
        System.out.println( greeting );
    }
}
[/CODE]

this ist eine Referenz auf das aktuelle Objekt selbst. Instanzvariablen gehören dauerhaft zum Objekt (sie existieren solange das Objekt existiert), Parameter hingegen sind nur während des Methodenaufrufs vorhanden.

Vollständiges Beispiel mit Konstruktor und mehreren Objekten

Eine Klasse funktioniert wie ein Keksausstecher – man kann damit beliebig viele Objekte erstellen:

[CODE]
class HelloObject
{
    private String greeting;

    public HelloObject( String greeting )
    {
        this.greeting = greeting;
    }

    public void speak()
    {
        System.out.println( greeting );
    }
}

public class HelloTester
{
    public static void main( String[] args )
    {
        HelloObject object1 = new HelloObject("Mercury");
        HelloObject object2 = new HelloObject("Venus");
        HelloObject object3 = new HelloObject("Earth");
        HelloObject object4 = new HelloObject("Mars");

        object1.speak();
        object2.speak();
        object3.speak();
        object4.speak();
    }
}
[/CODE]

Ausgabe:

Mercury
Venus
Earth
Mars

Jedes Objekt hat seine eigene Instanzvariable greeting mit einem anderen Wert. Die Klassendefinition existiert nur einmal; die Objekte werden zur Laufzeit aus ihr erstellt.

Zusammenfassung

Kapitel 46 legt das Fundament für das objektorientierte Programmieren in Java. Man lernt, dass eine Klasse ein Bauplan ist und ein Objekt eine konkrete Instanz dieses Bauplans. Klassen bestehen aus Instanzvariablen (Daten), Konstruktoren (Initialisierung) und Methoden (Verhalten). Das Schlüsselwort this hilft dabei, Instanzvariablen eindeutig von gleichnamigen Parametern zu unterscheiden. Die Driver-Klasse mit main() startet das Programm und koordiniert die Zusammenarbeit der Objekte.


Kapitel 47 – Klassendesign am Beispiel: Kegel (Cone)

Dieses Kapitel zeigt Schritt für Schritt, wie man eine eigene Klasse sinnvoll entwirft – von der Planung (welche Daten brauche ich?) über den Konstruktor bis zu Getter-, Setter- und Berechnungsmethoden. Als Beispiel dient ein gerader Kreiskegel, der vollständig durch Radius und Höhe beschrieben wird. Alle weiteren Eigenschaften (Oberfläche, Volumen) lassen sich daraus berechnen.
Instanzvariablen: private double radius und private double height – beide private, damit kein direkter Zugriff von außen möglich ist.
Getter & Setter: Kontrollierter Zugriff auf die privaten Felder über getRadius(), getHeight(), setRadius() und setHeight().
Berechnungsmethoden:

•	area() – berechnet die Gesamtoberfläche (Mantelfläche + Grundfläche) mit der Mantellinie s = √(r² + h²)

•	volume() – berechnet das Volumen mit V = π · r² · h / 3

[CODE]
public class Cone {
    private double radius;
    private double height;

    public Cone(double radius, double height) {
        this.radius = radius;
        this.height = height;
    }

    public double getRadius() { return radius; }
    public double getHeight() { return height; }

    public void setRadius(double radius) { this.radius = radius; }
    public void setHeight(double height) { this.height = height; }

    // Oberfläche: Mantelfläche + Grundfläche
    public double area() {
        double s = Math.sqrt(radius * radius + height * height); // Mantellinie
        return Math.PI * radius * radius + Math.PI * radius * s;
    }

    // Volumen
    public double volume() {
        return Math.PI * radius * radius * height / 3.0;
    }
}
[/CODE]

Beispielaufruf in einer Tester-Klasse:

Cone meinKegel = new Cone(3.0, 5.0);
System.out.println("Volumen: " + meinKegel.volume());
System.out.println("Oberfläche: " + meinKegel.area());

Merke: Objektorientiertes Design beginnt mit der Frage: Welche Daten gehören zum Objekt? Alles andere (Berechnungen, Ausgaben) ergibt sich daraus. Getter und Setter sind der sichere Weg, auf private Felder zuzugreifen.



Kapitel 49 – Klassendesign am Beispiel: Girokonto (CheckingAccount)

Dieses Kapitel zeigt OOP anhand eines realen Bankkonto-Szenarios. Die Klasse CheckingAccount verwaltet Kontonummer, Kontoinhaber, Kontostand und einen internen Nutzungszähler. Wichtig: die private Hilfsmethode incrementUse() ist nicht von außen aufrufbar – sie wird nur intern von processDeposit() und processCheck() genutzt. Bei processCheck() wird automatisch eine Gebühr von 15 € berechnet, wenn der Kontostand unter 100.000 liegt.

[CODE]
public class CheckingAccount {
    private String accountNumber;   // Instanzvariablen
    private String accountHolder;
    private double balance;
    private double useCount = 0;

    public CheckingAccount(String accNumber, String holder, double start) {
        this.accountNumber = accNumber;
        this.accountHolder = holder;
        this.balance       = start;
    }

    public double getBalance() { return balance; }

    public void processDeposit(double amount) {
        incrementUse();
        balance = balance + amount;
    }

    public void processCheck(double amount) {
        double charge;
        incrementUse();
        if (balance < 100000)
            charge = 15;
        else
            charge = 0;
        balance = balance - amount - charge;
    }

    public String toString() {
        return "Account: " + accountNumber + "\tName: " + accountHolder +
               "\tBalance: " + balance + "\tUse Count: " + useCount;
    }

    private void incrementUse() {   // private – nur intern nutzbar
        useCount = useCount + 1;
    }
}

Tester-Klasse:

CheckingAccount bobsAccount  = new CheckingAccount("999", "Bob",  100);
CheckingAccount jillsAccount = new CheckingAccount("111", "Jill", 500);

bobsAccount.processCheck(50);
System.out.println(bobsAccount.toString());
// Bob hat 100 – 50 – 15 (Gebühr) = 35.0

jillsAccount.processDeposit(500);
jillsAccount.processCheck(100);
jillsAccount.processCheck(100);
jillsAccount.processDeposit(100);
System.out.println(jillsAccount.toString());
[/CODE]


Merke: private Methoden wie incrementUse() sind Hilfsmethoden für die interne Logik. Nur die public-Methoden bilden die Schnittstelle nach außen.


Kapitel 50 – Kapselung und Sichtbarkeitsmodifikatoren (Encapsulation)

Kapselung (Encapsulation) bedeutet: Instanzvariablen werden mit private vor direktem Zugriff von außen geschützt. Der Zustand eines Objekts kann nur über definierte public-Methoden verändert werden – genau wie bei einem echten Bankkonto, auf das nur die Bank selbst Zugriff hat.

Modifier	Sichtbarkeit

public		Von überall zugreifbar
private		Nur innerhalb der eigenen Klasse
(kein Modifier)	Nur im gleichen Package

Zugriffsmethoden (Access Methods):

•	Getter (getX()) – liefern den Wert einer privaten Variable zurück

•	Setter (setX()) – setzen den Wert kontrolliert (können Validierung enthalten)

Das Girokonto aus Kapitel 49 ist ein perfektes Beispiel für Kapselung: Alle Felder sind private, der Kontostand kann nicht direkt von außen verändert werden, sondern nur über processDeposit() und processCheck(). Auch incrementUse() ist private – sie ist ein internes Implementierungsdetail.

[CODE]
// FALSCH – direkter Zugriff auf private Variable:
// bobsAccount.balance = 99999;  // Compilerfehler!

// RICHTIG – über public-Methode:
bobsAccount.processDeposit(500);
System.out.println(bobsAccount.getBalance());

Aus dem Repo (EqualsDemo.java) – Objektvergleich mit equals():

Point pointA = new Point(7, 99);
Point pointB = new Point(7, 99);  // gleiche Daten, anderes Objekt

if (pointA.equals(pointB))
    System.out.println("The two objects contain the same data: " + pointA);
else
    System.out.println("The two objects are not equivalent: " + pointA +
                       " differs from " + pointB);
[/CODE]

Merke: private schützt den Zustand eines Objekts vor ungewollten Änderungen von außen. Gute Kapselung macht Code sicherer, wartbarer und weniger fehleranfällig – ein zentrales Prinzip in der professionellen Softwareentwicklung.


Kapitel 51 – Parameter, Überladung und lokale Variablen

Methoden arbeiten mit zwei Arten von Variablen, die nicht dauerhaft im Objekt gespeichert werden:

•	Parameter (formale Parameter): Werden beim Methodenaufruf von außen übergeben. Sie existieren nur während der Ausführung der Methode.

•	Lokale Variablen: Werden innerhalb einer Methode deklariert und sind nur dort sichtbar. Nach dem Methodenende werden sie verworfen.

Ein gutes Beispiel aus dem Repo ist processCheck() in CheckingAccount.java – amount ist ein Parameter, charge ist eine lokale Variable:

[CODE]
public void processCheck(double amount) {   // amount = Parameter
    double charge;                           // charge = lokale Variable
    incrementUse();
    if (balance < 100000)
        charge = 15;
    else
        charge = 0;
    balance = balance - amount - charge;
}
[/CODE]

Methoden-Überladung (Overloading): 

Mehrere Methoden dürfen denselben Namen haben, solange sich ihre Signatur (Parameteranzahl oder -typen) unterscheidet. Java erkennt automatisch, welche Variante aufgerufen werden soll.

[CODE]
// Überladung: gleicher Name, verschiedene Parametertypen
public double area(double radius) {
    return Math.PI * radius * radius;
}
public double area(double laenge, double breite) {
    return laenge * breite;
}
[/CODE]

Die Signatur einer Methode besteht aus Name + Parameterliste. Der Rückgabetyp allein reicht nicht zur Unterscheidung. Lokale Variablen und Parameter mit gleichem Namen wie eine Instanzvariable verdecken diese – deshalb ist this.variablenname wichtig.
Merke: Lokale Variablen und Parameter sind kurzlebig – sie existieren nur während des Methodenaufrufs. Instanzvariablen gehören dauerhaft zum Objekt.




Kapitel 52 – Objekte als Parameter (Object Parameters)

Wenn ein primitiver Datentyp (z. B. int, double) an eine Methode übergeben wird, wird eine Kopie des Wertes übergeben (Call by Value). Änderungen innerhalb der Methode haben keinen Einfluss auf die ursprüngliche Variable.

Wenn ein Objekt als Parameter übergeben wird, wird eine Kopie der Referenz übergeben – nicht das Objekt selbst. Die Methode zeigt dann auf dasselbe Objekt im Speicher. Das bedeutet:

•	Bei unveränderlichen Objekten (z. B. String) kann die Methode das Original nicht verändern.

•	Bei veränderbaren Objekten (z. B. selbst definierte Klassen) kann die Methode das Originalobjekt tatsächlich verändern.

Aus EqualsDemo2.java – Referenzvergleich mit ==:

[CODE]
Point pointA = new Point(7, 99);   // erstes Objekt
Point pointB = new Point(7, 99);   // zweites Objekt – gleiche Daten, andere Referenz

if (pointA == pointB)
    System.out.println("The two variables refer to the same object");
else
    System.out.println("The two variables refer to different objects");
[/CODE]

// Ausgabe: "The two variables refer to different objects"
== vergleicht bei Objekten die Referenz (Speicheradresse), nicht den Inhalt. Für Inhaltsvergleich muss .equals() verwendet werden.

Aus EqualsTester.java – String-Objekte als Parameter:

[CODE]
public static boolean myEquals(String strA, String strB) {
    if (strA.isEmpty() && strB.isEmpty())   return true;
    else if (strA.isEmpty() || strB.isEmpty()) return false;
    else if (strA.charAt(0) != strB.charAt(0)) return false;
    else return myEquals(strA.substring(1), strB.substring(1));
}

String strA = "applecart";
String strB = "appleCart";
[/CODE]

// Strings sind unveränderlich – die Methode kann die Originale nicht verändern

Merke: Primitive Typen werden als Wert übergeben (sicher, keine Seiteneffekte). Objekte werden als Referenz übergeben – die Methode kann das Originalobjekt verändern, falls es nicht unveränderlich ist. String ist unveränderlich (immutable), eigene Klassen in der Regel nicht.





Kapitel 53 – Objekte, die andere Objekte enthalten

Komposition bedeutet: Eine Klasse enthält andere Objekte als Instanzvariablen. Genauso wie ein Fahrrad aus Rahmen, Rädern und Lenker besteht, kann ein Software-Objekt aus anderen Objekten zusammengesetzt sein. Das Kapitel zeigt das am Beispiel Tree (Baum = Cone als Krone + Cylinder als Stamm) und Forest (Wald = mehrere Tree-Objekte).

Cylinder.java – der Stamm eines Baumes:

[CODE]
public class Cylinder {
    private double radius;
    private double height;

    public Cylinder(double radius, double height) {
        this.radius = radius;
        this.height = height;
    }

    public double getRadius() { return radius; }
    public double getHeight() { return height; }
    public void setRadius(double radius) { this.radius = radius; }
    public void setHeight(double height) { this.height = height; }

    // Oberfläche: 2πr² + 2πrh
    public double area() {
        return 2.0 * Math.PI * radius * radius
             + 2.0 * Math.PI * radius * height;
    }

    // Volumen: πr²h
    public double volume() {
        return Math.PI * radius * radius * height;
    }
}
[/CODE]

Forest.java – ein Objekt, das mehrere Tree-Objekte enthält:

[CODE]
public class Forest {
    private Tree tree0 = null, tree1 = null,
                tree2 = null, tree3 = null;

    public void setTree(int treeNum, Tree tree) {
        if      (treeNum == 0) tree0 = tree;
        else if (treeNum == 1) tree1 = tree;
        else if (treeNum == 2) tree2 = tree;
        else if (treeNum == 3) tree3 = tree;
    }

    public Tree getTree(int treeNum) {
        if      (treeNum == 0) return tree0;
        else if (treeNum == 1) return tree1;
        else if (treeNum == 2) return tree2;
        else if (treeNum == 3) return tree3;
        else                   return null;
    }

    public String toString() {
        if (tree0 == null && tree1 == null &&
            tree2 == null && tree3 == null)
            return "No Trees in Forest";
        String str = "Forest:\n";
        if (tree0 != null) str += "Tree 0: " + tree0.toString() + "\n";
        if (tree1 != null) str += "Tree 1: " + tree1.toString() + "\n";
        if (tree2 != null) str += "Tree 2: " + tree2.toString() + "\n";
        if (tree3 != null) str += "Tree 3: " + tree3.toString() + "\n";
        return str;
    }
}
[/CODE]

ForestTester.java – Verwendung der komponierten Objekte:

[CODE]
Forest myForest = new Forest();

double trunkR = 1.0, trunkH = 1.0;
double branchR = 8.0, branchH = 10.0;

Tree tree = new Tree(trunkR, trunkH, branchR, branchH, 1, 2, 3);
myForest.setTree(0, tree);

branchR = 15.0; branchH = 30.0;

[/CODE]

Merke: Komposition ist eines der mächtigsten OOP-Konzepte. Komplexe Objekte werden aus einfacheren Bauteilen zusammengesetzt. Wenn eine Klasse ein anderes Objekt enthält, sollte dieses ebenfalls mit sinnvollen Getter/Setter-Methoden ausgestattet sein.



Kapitel 54 – Statische Methoden (static)

Normale Instanzmethoden gehören zu einem bestimmten Objekt und können auf dessen Instanzvariablen zugreifen. Statische Methoden gehören zur Klasse selbst – sie benötigen kein Objekt und werden direkt über den Klassennamen aufgerufen.

Wann static verwenden?

•	Wenn die Methode keine Instanzvariablen benötigt

•	Hilfsmethoden und Berechnungen, die unabhängig von einem Objekt sind

•	Bekanntestes Beispiel: Math.sqrt(), Math.PI, Integer.parseInt()
static main() ist immer statisch, weil sie vor der Erstellung jedes Objekts ausgeführt wird – es gibt zu diesem Zeitpunkt noch kein Objekt.

Aus EvalSqrt.java – Nutzung von Math.sqrt(), einer statischen Methode:

[CODE]
System.out.println("Square root of " + x + " is " + Math.sqrt(x));
Eigene statische Methode – Beispiel:
java
public class Rechner {
    public static double harmonischeSumme(int n) {
        double sum = 0.0;
        for (int term = 1; term <= n; term++)
            sum += 1.0 / term;
        return sum;
    }
}

// Aufruf direkt über Klassenname – kein Objekt nötig:

System.out.println(Rechner.harmonischeSumme(6));

HarmonicTester.java – harmonische Reihe berechnen:

int term = 1, lastTerm = 6;
double sum = 0.0;

while (term <= lastTerm) {
    sum += 1.0 / term;
    term++;
}
System.out.println("Sum of 6 terms: " + sum);
[/CODE]

Wichtige Regel: Eine statische Methode kann keine Instanzvariablen oder Instanzmethoden aufrufen – sie hat keinen Zugriff auf this.
				Instanzmethode		Statische Methode
Benötigt Objekt?		Ja			Nein
Zugriff auf Instanzvariablen?	Ja			Nein
Aufruf				objekt.methode()	Klasse.methode()

Merke: static ist sinnvoll für Hilfsmethoden ohne Objektbezug. Die Math-Klasse ist das beste Beispiel – sie hat nur statische Methoden und wird nie instanziiert.
